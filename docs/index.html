What It Does

The automated QA lead who actually reads the spec

You give it a URL, a PRD, and your codebase. It comes back with a prioritized list of everything that is incomplete, broken, or missing. Each finding is verified, critiqued for false positives, and linked to the specific PRD requirement it violates.

🔍

Reads Your PRD

Parses features, acceptance criteria, user flows, and priorities from your product requirements document.

💻

Analyzes Your Code

Scans routes, forms, auth patterns, and framework structure. Supports Next.js, Express, React Router, and more.

🌐

Visits Your App

Uses Playwright to crawl pages, fill forms, test viewports, and capture screenshots in a real browser.

📄

Files GitHub Issues

Creates prioritized issues with evidence, reproduction steps, and PRD traceability. Deduplicates against existing issues.

Quick Start

Up and running in 60 seconds

Prerequisites

  • Node.js 18+
  • ANTHROPIC_API_KEY environment variable set
  • Your web application running locally
  • A PRD document (Markdown)

Install

# Install globally
npm install -g audit-agent

# Or run directly
npx audit-agent audit --url http://localhost:3000 --prd docs/prd.mdbash

Run Your First Audit

# Full audit with browser testing
npx audit-agent audit \
  --url http://localhost:3000 \
  --prd docs/prd.md \
  --codebase-path .

# Code-only audit (no browser needed)
npx audit-agent audit \
  --url http://localhost:3000 \
  --prd docs/prd.md \
  --browser none

# Resume a failed audit
npx audit-agent audit --resumebash
CLI Reference

Commands & Flags

audit

Run a full audit of the target application against its PRD.

npx audit-agent audit [options]

verify

Re-check a specific finding to see if it has been fixed.

# Verify by finding ID
npx audit-agent verify --finding F-001 --url http://localhost:3000

# Verify by GitHub issue number
npx audit-agent verify --issue 42 --url http://localhost:3000bash

How It Works

14-Phase Audit Pipeline

Each audit runs through 14 phases in sequence. Phases are categorized by execution type: pure TypeScript (no LLM), LLM-driven analysis (via Anthropic API), or browser+LLM (Playwright collects data, then Claude analyzes).

0 · Preflight Validate config, check connectivity, create audit directory
pure-ts
1 · PRD Parsing Extract features, acceptance criteria, priorities from your PRD
llm
2 · Code Analysis Scan routes, forms, auth patterns, framework detection
llm
3 · Progress Init Initialize progress tracking, set up artifact paths
pure-ts
4 · Safety Check Detect production environments, configure allow/deny lists, gate actions
pure-ts
5 · Exploration BFS route crawling, page capture, screenshot collection, SPA handling
browser+llm
6a · Form Testing Discover forms, fill inputs, test validation, capture before/after
browser+llm
6b · Responsive Testing Test mobile, tablet, desktop viewports. Detect overflow and layout breaks
browser+llm
7 · Finding Quality Critique findings for evidence, dedup, filter false positives
llm
8 · Report Generation Generate Markdown report + static HTML dashboard
pure-ts
9 · Interactive Review Optional human review of findings before filing issues
llm
10 · GitHub Issues Create issues with severity labels, screenshots, dedup against existing
pure-ts
11 · Verification Re-check each finding 3x to confirm reproducibility
browser+llm
12 · Polish Final cleanup, summary statistics, completion score
pure-ts
pure-ts (6 phases) llm-driven (4 phases) browser+llm (4 phases)
Configuration

config.yml

Place a config.yml in your project's .complete-agent/ directory to configure persistent settings. All options can also be set via CLI flags or environment variables. 

# .complete-agent/config.yml
version: "2.0"

# Target application
target:
  url: "http://localhost:3000"
  codebase_path: "."
  prd_path: "docs/prd.md"

# Authentication (optional)
auth:
  strategy: "cookie"
  cookies:
    - name: "session"
      value: "${AUTH_COOKIE}"
      domain: "localhost"

# LLM settings
llm:
  model: "claude-sonnet-4-5-20250929"
  max_budget_usd: 5.0

# Browser settings
browser:
  headless: true
  spa_strategy: "domstable"
  screenshots: true

# Exploration limits
exploration:
  max_pages: 50
  max_routes: 50

# Testing
testing:
  safe_mode: auto
  max_forms: 20
  test_viewports: [375, 768, 1280]

# Reporting
reporting:
  create_github_issues: true
  min_severity: "P2"yaml

CI Integration

GitHub Action

Run audits automatically on preview deployments. Findings are posted as PR comments.

# .github/workflows/completion-audit.yml
name: Completion Audit
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: paulrbrown/audit-agent@v2
        with:
          url: ${{ steps.deploy.outputs.url }}
          prd: docs/prd.md
          anthropic-api-key: ${{ secrets.ANTHROPIC_API_KEY }}yaml
The Vision

Why this exists

Every "audit tool" finds problems. This one finds the right problems — the delta between what you said you would build and what you actually built. It is not a linter, not a test runner, not a security scanner. It is the automated version of the QA lead who reads the spec and then actually uses the product.

Who it's for

Solo developers and small teams (2-5 engineers) building web applications who want to ship complete work. The person who knows the PRD exists but hasn't had time to check every requirement against the running app.

How to use it

Run it before every release. Point it at your staging deployment. Review the findings. Close the issues. Ship with confidence that nothing was missed.

What's next

  • Continuous completion tracking via PR comments
  • Multi-PRD support for large projects
  • Regression detection across audit runs
  • Visual regression via screenshot diffing
  • Learning from rejected findings to reduce false positives
